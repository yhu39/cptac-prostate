# CPTAC Prostate Repository Architecture Summary

## Scope of this repository

python env: C:\Users\yhu39\AppData\Local\anaconda3\envs\prostate

This repository does not implement raw-spectrum peptide or glycopeptide search. It starts from already-produced result tables generated by external tools such as MSPyCloud and MSFragger, then performs metadata cleaning, sample harmonization, quality control, PCA, differential analysis, and downstream result export.

The actual packaged Python code is small and centered on config-driven table processing:

- `src/cptac_prostate/cli.py` dispatches only `clean_meta` and `clean_data`.
- `src/cptac_prostate/clean_meta.py` contains the main implemented workflow.
- `src/cptac_prostate/clean_data.py` is largely a scaffold with placeholder steps.
- `src/cptac_prostate/20260401_mspycloud_Global_DDA_tumor_vs_normal.py` is an exported notebook-style downstream analysis script.

## Current pipeline from raw spectra to final results

The effective pipeline is:

1. Raw spectra are searched and quantified outside this repository by MSPyCloud and/or MSFragger.
2. Those external tools produce protein-level abundance or log2-ratio matrices plus sample-ID mapping files.
3. Metadata is cleaned and harmonized:
   - sample IDs are renamed, for example `C3L.05292.T -> C3L-05292_T`
   - tumor and normal subsets are extracted
   - Gleason grade dictionaries/tables/plots are created
4. MSPyCloud data is cleaned:
   - load `protein_matrix-log2_ratios-MD_norm.tsv`
   - rename sample columns using the Excel sample map
   - remove contaminants matching `Cont|`
   - median-center samples
   - write `mspycloud_Global_DDA_TMT_Clean_MD.tsv`
5. QC notebooks compare metadata sample sets against data matrix columns, often using Venn diagrams.
6. Search-engine comparison notebook compares MSPyCloud and MSFragger outputs per sample using Pearson correlation, RMSE, and Bland-Altman plots.
7. PCA notebooks subset samples by tumor/normal, purity, stage, or grade group and plot PCA from no-missing matrices.
8. Differential notebooks run group comparisons using Wilcoxon tests with missingness thresholds, FDR cutoff, and log2 fold-change thresholds.
9. Significant genes are exported to text files and optionally used for enrichment and heatmap workflows.

Architecturally, the repository begins at the stage of "post-search quantified protein matrix plus metadata" rather than "raw spectra".

## Where quantitative analysis could be inserted

### Best insertion point 1: upstream of this repository

If the goal is peptide-level or glycopeptide-level quantitative analysis from raw spectra or search evidence, that belongs in a new upstream module before the current workflows begin.

That new layer would:

- ingest raw search outputs or peptide/PSM evidence
- perform quantification, filtering, aggregation, and normalization
- emit a matrix compatible with the current downstream notebooks

### Best insertion point 2: inside the current data-clean stage

The cleanest existing in-repo hook is the matrix-cleaning stage:

- `notebooks/QualityControl/20260401_CPTAC_mspycloud_Global_DDA_TMT_DataClean.ipynb`
- `src/cptac_prostate/clean_data.py`

This is the natural place for:

- extra normalization
- batch correction
- imputation
- peptide-to-protein rollup
- glycopeptide aggregation
- reporter-ion summarization

### Best insertion point 3: before PCA and differential analysis

If quantification is already done externally and only post-quant processing is needed, insert it after cleaned matrix generation and before:

- PCA notebooks using `plot_pca1`
- differential notebooks using `compare_two_groups`

## Main conclusion

- There is currently no peptide/glycopeptide search engine in the repository.
- There is currently no in-repo spectrum-level scoring/FDR engine.
- The repository is primarily a downstream post-processing and analysis layer over externally generated proteomics quant tables.
