# CPTAC Prostate Repository Architecture

## Scope of this repository

This repository does not implement raw-spectrum peptide or glycopeptide search. It starts from already-produced result tables generated by external tools such as MSPyCloud and MSFragger, then performs metadata cleaning, sample harmonization, quality control, PCA, differential analysis, purity-sensitive validation, and downstream result export.

The packaged Python code is centered on config-driven table processing:

- `src/cptac_prostate/cli.py` dispatches `clean_meta`, `clean_data`, `purity_test`, and `purity_validation`
- `src/cptac_prostate/clean_meta.py` contains the main implemented metadata workflow
- `src/cptac_prostate/clean_data.py` is largely a scaffold with placeholder steps
- `src/cptac_prostate/purity_validation.py` contains the current reproducible purity-validation workflow
- `src/cptac_prostate/20260401_mspycloud_Global_DDA_tumor_vs_normal.py` is an exported notebook-style downstream analysis script

## Modules responsible for each area

### Peptide/glycopeptide search

There is no in-repo module that performs peptide search, glycopeptide search, raw spectra parsing, PSM generation, or spectrum-level FDR estimation.

Evidence:

- no meaningful `src/` code references to glycopeptide search logic, mzML/mzXML parsing, spectra handling, or PSM processing
- notebooks load finished search and quantification outputs from external paths

Relevant ingest notebooks:

- `notebooks/QualityControl/20260401_CPTAC_mspycloud_Global_DDA_TMT_DataClean.ipynb`
- `notebooks/QualityControl/20260401_CPTAC_Global_DDA_comparison.ipynb`
- `notebooks/Differential/20260401_msfragger_Global_DDA_TMT_Diff.ipynb`

### Scoring

Search-engine scoring is external to this repo. The scoring implemented here is downstream statistical scoring and QC comparison:

- differential testing through two-group comparisons such as Wilcoxon-based workflows
- volcano-plot significance categorization
- cross-search-engine comparison via correlation and RMSE
- purity-adjusted linear modeling in the purity workflows

Primary locations:

- `src/cptac_prostate/global_diff.py`
- `src/cptac_prostate/purity_test.py`
- `src/cptac_prostate/purity_validation.py`
- notebooks in `notebooks/Differential/`
- notebooks in `notebooks/QualityControl/`

## Effective pipeline from quant table to final results

1. Raw spectra are searched and quantified outside this repository by MSPyCloud and/or MSFragger.
2. External tools produce protein-level abundance or log2-ratio matrices plus metadata and sample-ID mapping files.
3. Metadata is cleaned and harmonized.
4. Protein matrices are cleaned and normalized.
5. QC notebooks compare metadata sample sets against data matrix columns.
6. Search-engine comparison notebooks compare MSPyCloud and MSFragger outputs.
7. PCA workflows evaluate separation by tumor/normal, purity, stage, or grade group.
8. Differential workflows perform group comparisons and export significant proteins.
9. Enrichment workflows use significant or ranked protein lists for pathway interpretation.
10. Purity-validation workflows test whether protein-level and pathway-level signals attenuate after purity adjustment.

Architecturally, the repository begins at the stage of "post-search quantified protein matrix plus metadata" rather than raw spectra.

## Where quantitative analysis could be inserted

### Best insertion point 1: upstream of this repository

If the goal is peptide-level or glycopeptide-level quantitative analysis from raw spectra or search evidence, that belongs in a new upstream module before the current workflows begin.

That new layer would:

- ingest raw search outputs or peptide/PSM evidence
- perform quantification, filtering, aggregation, and normalization
- emit a matrix compatible with the current downstream workflows

### Best insertion point 2: inside the current matrix-clean stage

The cleanest existing in-repo hook is the matrix-cleaning stage:

- `notebooks/QualityControl/20260401_CPTAC_mspycloud_Global_DDA_TMT_DataClean.ipynb`
- `src/cptac_prostate/clean_data.py`

This is the natural place for:

- extra normalization
- batch correction
- imputation
- peptide-to-protein rollup
- glycopeptide aggregation
- reporter-ion summarization

### Best insertion point 3: before PCA and downstream modeling

If quantification is already done externally and only post-quant processing is needed, insert it after cleaned matrix generation and before:

- PCA workflows
- differential analysis
- purity-validation workflows

## Main conclusion

- There is no peptide/glycopeptide search engine in the repository.
- There is no in-repo spectrum-level scoring or FDR engine.
- The repository is primarily a downstream post-processing and analysis layer over externally generated proteomics quant tables.
- The current in-repo statistical workflows include differential testing and purity-sensitive validation on those downstream matrices.
